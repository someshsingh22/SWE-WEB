Benchmarks for autonomous coding agents overwhelmingly evaluate functional correctness: resolving GitHub issues against predefined test cases \citep{jimenez2024swebenchlanguagemodelsresolve}, synthesizing functions from specifications \citep{chen2021codex,austin2021programsynthesislargelanguage}. Real software quality, however, extends beyond correctness to non-functional properties (performance, stability, responsiveness) that users experience but test suites do not capture. As coding agents move from research prototypes to deployed tools used by over 84\% of developers\footnote{\url{https://survey.stackoverflow.co/2025/}}, whether they can optimize these user-facing properties under rigorous, execution-based evaluation remains an open question.

We study this question through web performance: \emph{can autonomous coding agents improve user-facing Core Web Vitals (CWV; see \S\ref{sec:background_cwv}) without degrading other dimensions of experience?} We introduce \OurBench, a benchmark that evaluates agents on optimizing three CWV metrics---Largest Contentful Paint (LCP, perceived loading speed), Cumulative Layout Shift (CLS, visual stability), and Interaction to Next Paint (INP, responsiveness)---for real websites sourced from production GitHub repositories.

Existing coding-agent benchmarks occupy different evaluation axes. The SWE-bench ecosystem \citep{jimenez2024swebenchlanguagemodelsresolve} tests functional correctness on Python repositories via unit tests; its derivatives (SWE-bench Verified, SWE-bench M, SWE-bench Pro \cite{deng2025swe}, SWE-smith, SWE-rebench) inherit this scope. GSO \cite{XXX_gso}, the closest prior work, also evaluates a non-functional property, runtime speed, via execution-based measurement, but on systems codebases using performance tests as the specification, with tasks derived from developer commit histories. BaxBench \cite{XXX_baxbench} evaluates correctness and security of LLM-generated backends, where agents generate applications from scratch rather than editing existing repositories. \OurBench is orthogonal: it measures \emph{user-facing browser experience} on \emph{existing production web repositories} via a \emph{real browser}, not a unit test, performance microbenchmark, or static analyzer. The distinctions from GSO matter concretely: \OurBench measures browser-level user experience rather than computational throughput; measurement requires hosting the patched site and rendering it in a browser, not executing a performance test; success requires multi-metric Pareto improvement across mobile and desktop, so agents cannot win by improving one metric at the cost of another; and tasks are real production web repositories evaluated end-to-end, not optimization targets extracted from commit histories. Moreover, while the SWE-bench ecosystem evaluates exclusively Python repositories, web performance optimization operates over JavaScript, TypeScript, HTML, and CSS, which constitute over 40\% of GitHub\footnote{\url{https://github.blog/news-insights/octoverse/}} yet remain unrepresented in coding-agent benchmarks.

CWV provides a controlled testbed for this evaluation because it gives standardized, browser-level signals that are objective, reproducible, and independent of unit-test design. The three metrics interact in ways that demand multi-objective reasoning: deferring JavaScript may improve INP but delay LCP; async CSS loading may speed rendering but cause layout shifts that degrade CLS. Effective optimization therefore requires coordinated changes across assets, configuration, and application logic (median patches in our evaluation modify 3--5 files) while preserving metrics that are already healthy. CWV improvements also carry direct economic relevance: a Deloitte study reports that a 0.1\,s reduction in LCP increases conversions by up to 10\%\footnote{\url{https://www.deloitte.com/ie/en/services/consulting/research/milliseconds-make-millions.html}}.

\OurBench comprises 2{,}741 GitHub repositories and 73{,}000 deduplicated webpages across 17 web frameworks (including React, Vue, Next.js, Hugo, and Jekyll), with a scalable curation pipeline covering 10{,}700+ candidate repositories. Each instance consists of a pinned repository snapshot, a deterministic containerized deployment, and a set of webpages measured for CWV. Agents may freely modify the codebase; success requires a \emph{Pareto improvement}: at least one CWV metric improves by more than a noise margin (calibrated from repeated measurement variance; \S\ref{sec:eval-metrics}) with no metric degrading beyond that margin, across both mobile and desktop device profiles. This Pareto criterion is new among coding-agent benchmarks: no existing benchmark uses multi-metric, multi-device joint scoring, and it prevents agents from winning by improving one metric at the cost of another. Because CWV is measured dynamically from deployed pages rather than matched against reference patches, the benchmark structurally resists data contamination: patch memorization provides no direct advantage, since the agent must produce changes that improve browser-measured metrics on the live hosted site. The curation pipeline continuously incorporates new repositories ($\sim$670 per month, of which $\sim$280 have Chrome User Experience Report field data), further mitigating saturation.

We evaluate eleven agent--LLM configurations on a stratified subset of 300 websites (covering Poor, Needs Improvement, and Good baseline CWV profiles) under a structured plan-then-implement protocol: a read-only planning phase produces a structured optimization plan, followed by a write-only execution phase (\S\ref{sec:experiments}). Each site is measured over 30 independent browser runs per device profile. The best configuration (OpenCode + GPT-5.1\,Codex) achieves Pareto CWV improvement on 56.7\% of sites with a 20\% degradation rate; a strong middle tier includes Gemini\,2.5\,Pro (53.6\%) and Claude~Code with Sonnet\,4.6 (51.7\%). The weakest configuration (OpenCode + GPT-4.1) achieves 0\% Pareto improvement while degrading 90\% of sites. Controlled ablations separate two factors: the agent scaffold governs multi-metric planning (a 16$\times$ Pareto-rate gap between the best and worst frameworks at fixed model), while the underlying LLM determines whether the plan executes without collateral regressions. Vanilla prompting---without the structured protocol---yields 0\% improvement with any agent; gains emerge only under the plan-then-implement protocol.

We highlight the following contributions:
\begin{enumerate}
    \item We introduce \OurBench, the first execution-based benchmark that evaluates coding agents on improving user-facing CWV for real websites (live GitHub Pages sites verified against their public renderings), covering 2{,}741 repositories, 73{,}000 webpages, and 17 frameworks, with a refreshable pipeline scalable to 10{,}700+ repositories.

    \item We design a framework-agnostic, containerized evaluation harness that deploys websites, measures CWV under controlled conditions, and enforces visual regression, build integrity, and content preservation checks to prevent reward hacking.

    \item We introduce a Pareto success criterion over six metric-device combinations (LCP, CLS, INP $\times$ mobile, desktop) that prevents single-metric gaming and enforces joint improvement, a scoring mechanism not used by any existing coding-agent benchmark.

    \item We evaluate eleven agent--LLM configurations, revealing that the best achieve Pareto improvement on 57\% of sites while the worst degrade 90\%, with models remaining particularly poor at improving CLS and INP. Controlled ablations show that scaffold design and model capability contribute in separable, additive ways to multi-objective optimization.
\end{enumerate}
